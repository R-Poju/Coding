1、Protobuf编译器（protoc）

作用：将.proto文件翻译成目标语言代码（如C++的.pb.h/.pb.cc）

核心功能：

生成消息结构体（如GetVarifyReq），包含字段的getter/setter方法和序列化/反序列化逻辑（自动处理二进制编码）

生成RPC服务骨架（如VarifyService::Service）和客户端存根（如VarifyService::Stub）



2、gRPC框架

作用：基于protobuf的RPC通信框架，负责网络传输（HTTP/2）、连接管理、序列化/反序列化等底层工作

优势：跨语言（C++/Java/Go/Python等）、高性能（二进制编码+HTTP/2多路复用）、自动生成代码减少手写逻辑



3、序列化与反序列化

序列化：客户端调用时，将GetVarifyReq结构体转换成二进制数据（通过protobuf自动实现）

反序列化：服务端收到二进制数据后，自动转化成GetVarifyReq结构体（通过protobuf自动实现）

好处：比JSON/XML更高效（体积小，解析快），并且跨语言兼容



除了gRPC，protobuf还可用于普通数据存储（如配置文件、日志）或非RPC通信（如消息队列）





| **阶段**          | **作用**                                              | **开发者工作**                                               |
| ----------------- | ----------------------------------------------------- | ------------------------------------------------------------ |
| **定义 `.proto`** | 制定“通信契约”（数据结构+接口方法）                   | 用 `message`定义数据，`service`定义 RPC 方法，明确参数和返回值。 |
| **生成代码**      | 自动生成序列化/反序列化代码、RPC 服务骨架、客户端存根 | 执行 `protoc`命令，生成目标语言代码（如 C++ 的 `.pb.h`/`.grpc.pb.h`）。 |
| **服务端实现**    | 填充业务逻辑（如数据库操作、邮件发送、消息推送）      | 继承生成的 `Service`类，实现 `rpc`方法，处理请求并返回响应。 |
| **客户端调用**    | 发起 RPC 请求，获取响应                               | 用生成的 `Stub`类调用方法，像调用本地函数一样简单。          |

通过这个流程，`.proto`文件成为**跨团队、跨语言的“沟通桥梁”**：后端团队按它实现服务，前端/客户端团队按它调用接口，无需反复确认“参数怎么传”“返回什么格式”，极大提升了开发效率。



传输过程：

1、客户端构造请求，创建对象，并按定义填充字段

2、gRPC自动将对象序列化为二进制流

3、服务端接收并解析请求，服务端收到二进制流后，gRPC会自动反序列化为对象。如果传输的数据不符合格式，反序列化会失败，服务端返回错误（如INVALID_ARGUMENT）

4、服务端处理业务逻辑，构造响应

5、传输，响应序列化，客户端解析







服务端能精准返回响应，核心是gRPC基于HTTP/2协议的多路复用和流（Stream）机制，每个RPC请求/响应对会被分配一个唯一的流ID，服务端通过流ID关联请求来源和响应目标，就像快递员通过运单号知道包裹要送回给谁



#### 1. **HTTP/2 流：请求的“专属通道”**

HTTP/2 允许在**同一个 TCP 连接**上同时传输多个“流”（Stream），每个流对应一个独立的请求/响应对。当你用 `stub->GetVarifyCode`发起调用时：

- 

  客户端 gRPC 框架会创建一个**新流**，分配一个**唯一流 ID**（如 `stream_id=5`）；

- 

  将请求数据（`GetVarifyReq`的二进制流）打上 `stream_id=5`的标签，通过 TCP 连接发送给服务端；

- 

  服务端收到后，根据 `stream_id=5`识别这是“客户端 A 的某个请求”，处理完后，将响应数据（`GetVarifyRsp`的二进制流）也打上 `stream_id=5`的标签，通过**同一个 TCP 连接**发回给客户端。

#### 2. **服务端的“流上下文”：知道“谁发的请求”**

在服务端处理请求时，gRPC 会为每个流创建一个 **`ServerContext`对象**（你代码中的 `::grpc::ServerContext* context`），它包含：

- 

  **流 ID**：隐式关联当前请求/响应的“专属通道”；

- 

  **客户端地址**：通过 `context->peer()`可获取客户端 IP:端口（用于日志或限流）；

- 

  **元数据（Metadata）**：客户端通过 `ClientContext`传递的附加信息（如认证 Token、请求 ID）。









上下文（`ClientContext`和 `ServerContext`）是 gRPC 中携带请求/响应附加信息的“信封”，用于在客户端和服务端之间传递“非业务数据”（如超时、认证、追踪 ID 等）。







stub（存根）是 gRPC 生成的客户端代理对象，封装了网络通信细节，让客户端调用 RPC 方法像调用本地函数一样简单。它的“来源”是 protoc编译器 + gRPC 插件根据 .proto文件自动生成的代码。
